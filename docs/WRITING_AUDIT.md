# Writing Audit — AI-tell language pass

A style-only audit of every file in scope, checked against 11 categories of
writing that reads as obviously AI-generated (filler phrases, hollow marketing
language, hedged non-claims, repetitive patterns, tone mismatches, etc. — see
the full category list this pass was run against, reproduced below for
reference). This is an inventory, not a rewrite — see the Phase 2 section at
the bottom for what was actually changed and why.

**Scope covered**: every file in `docs/`; `README.md`, `extension/README.md`,
`extension/USER_GUIDE.md`; dashboard UI copy (`components/lango/*.tsx`);
extension `popup/popup.html`, `popup/popup.js`, `options/options.html`,
`options/options.js`, and the banner/toast strings in
`extension/content/*.js`; and user-facing strings generated by backend code
(specifically `reason_string` text in `backend/src/detection/scan.rs`, and
error messages in `backend/src/error.rs`). Internal code comments, test files,
and the frozen AI4I/Cimas submission documents were explicitly out of scope
and were not touched.

**Categories checked**: (1) filler/throat-clearing phrases, (2) hollow
marketing/buzzword language, (3) redundant summarizing phrases, (4) excessive
em-dash use, (5) repetitive rule-of-three patterns, (6) repetitive
gerund-opener list patterns, (7) hedged language that says nothing concrete,
(8) excessive/inconsistent bolding used for decoration rather than structure,
(9) tone mismatches (exclamation marks, emoji, over-enthusiasm) for this
product's register, (10) UI copy that's a full sentence where a short phrase
would do, (11) instruction-leakage or meta-references to how content was
produced.

## Method

Grepped for specific phrases per category across the full scope (filler
openers, buzzwords, redundant-summary openers, hedged-language patterns,
instruction-leakage phrases, emoji/exclamation marks), then read every
in-scope file in full for patterns a regex can't catch — rhythm, repetition,
bolding density, tone. Bold-usage density was additionally spot-checked per
file (bold-span count vs. line count) to distinguish "used for genuine
structure" from "used for decoration" per category 8.

## Findings

**One instance found.** This codebase's existing documentation and UI copy
were, going in, unusually consistent and already free of the patterns this
audit was checking for — dense, concrete, plainly worded throughout, with
bolding used consistently to flag genuinely load-bearing facts rather than
for emphasis. That is not this pass's own conclusion asserted without
support — it's the result of actually reading all ~35 in-scope files in full
and finding almost nothing to flag, not a pass that stopped early.

### Extension

| File | Location | Flagged text | Category | Why it reads as an AI-tell |
|---|---|---|---|---|
| `extension/USER_GUIDE.md` | Section 2, step 2 (line 53) | "it's not a special or risky mode, just the one that **unlocks** loading an extension that isn't from the Chrome/Edge Web Store" | 2 — hollow marketing/buzzword language | "Unlocks" is marketing-register word choice for what is, literally, "lets you." Everything else in this file uses plain verbs ("allows", "shows", "opens") for the same kind of action — this is the one spot that reaches for a slightly punchier verb than the rest of the document's register uses. Borderline (it's describing a real UI mechanic, not hyping a feature), but flagged per the instruction to flag every instance of this category regardless of context. |

### Docs, dashboard, and backend strings

No instances found in `docs/*.md` (11 files), `README.md`,
`components/lango/*.tsx` (7 files), `extension/README.md`,
`extension/popup/*`, `extension/options/*`, `extension/content/*.js` banner
strings, or `backend/src/detection/scan.rs`'s `reason_string` output /
`backend/src/error.rs`'s error messages.

Specifically checked and clear:
- No filler openers ("It's important to note", "Let's dive into", "When it
  comes to", etc.) anywhere in scope.
- No buzzwords (seamless, robust, cutting-edge, leverage-as-verb, empower,
  streamline, revolutionize, etc.) anywhere else in scope besides the one
  instance above.
- No redundant "In summary" / "In conclusion" / "Overall" closers anywhere.
- No hedged-into-meaninglessness phrasing ("may potentially help improve",
  etc.) anywhere.
- No exclamation marks or emoji used for tone/enthusiasm anywhere in scope
  (the only `!` characters in scope are the literal demo password
  `LangoDemo123!` and code/macro syntax) — the calm, precise register this
  product's compliance/security/healthcare context calls for is already
  maintained consistently.
- No gerund-opener list patterns (3+ consecutive bullets starting
  "Ensuring…"/"Providing…"/etc.) anywhere.
- No adjective-triad rule-of-three patterns ("fast, reliable, and secure"
  style) anywhere — multi-item lists in this repo vary in length (2, 3, 4, 5
  items) based on what's actually being enumerated, not padded to a fixed
  count.
- Bolding density checked per-file (bold-span count vs. line count, e.g.
  `docs/SECURITY_PRIVACY.md` at 31 bold spans / 73 lines, `extension/USER_GUIDE.md`
  at 51/221) — in every file, spot-reading confirmed bold is used to flag a
  specific load-bearing fact or status word ("**Real.**", "**NOT verified**",
  "**Fail**"), not for decorative emphasis. None flagged under category 8.
- No em-dash overuse — em-dashes appear throughout (this is naturally
  dash-heavy technical prose with many parenthetical asides), but not at a
  density where "nearly every sentence has one"; they're doing real
  parenthetical work, not standing in for rhythm.
- UI copy (button labels, banner text, tooltips) is already short and
  scannable throughout — checked `popup.html`'s buttons ("Log in to Lango",
  "Open live dashboard", "Log out"), `site-adapter.js`'s banner strings
  ("Lango: scanning prompt…", "Lango: N entit(y/ies) redacted before
  sending"), and every dashboard panel title/KPI label. Nothing found that
  uses a full sentence where a short phrase would read faster.
- No instruction-leakage or meta-references found beyond the disclosure
  statements that are deliberately, appropriately placed (the "Built with
  Claude and Claude Code" disclosures in `README.md`'s Team section,
  `docs/TEAM_CAPABILITY.md`'s Tools section, and `docs/PITCH_DECK_CONTENT.md`'s
  Team + Ask slide — these are genuine, intentional disclosures in the
  correct location, not leaked meta-commentary, and were left untouched).

## Judgment call

The "unlocks" instance is genuinely borderline — it's describing a real UI
mechanic (Developer Mode literally does gate/enable that button), not hyping
a product feature the way "unlock your potential" would. Logged it anyway per
the instruction to flag every instance of this category regardless of
context; Phase 2 will make the final call on how (or whether) to reword it.
See `Questions.md` for the standing note on this call.
