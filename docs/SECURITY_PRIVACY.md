# Security and Privacy — Lango / AI Data Guard

| Area | Lango's answer |
|---|---|
| **Data minimisation** | **v0.1 (deployed)**: no real institutional data is collected — every prompt processed today is synthetic (seed-script-generated or manually tested), and the system never stores raw prompt text regardless: `backend/src/detection/scan.rs` redacts matched entities before anything is logged, and `audit_log.original_prompt_hash` stores only a SHA-256 hash of the original, never the text itself. This is real, running code, not a target-state description. **The native chat feature's `chat_messages` table follows the identical principle**: a user's message is stored only as its redacted form (never the raw original the person typed), and a blocked prompt creates no `chat_messages` row at all. The one place this differs from the prompt side: an assistant's reply is stored verbatim, since `scan_response` was never designed to redact a reply (see the Response scanning / Native chat sections of [ARCHITECTURE.md](ARCHITECTURE.md)) and this chat surface — unlike the browser extension, which never needs to persist a reply already rendered on a third-party page — is the only place responsible for redisplaying it later. **Target**: same principle at institutional scale — the Redaction Engine strips sensitive entities before a prompt is forwarded to an AI provider; the audit log records that a redaction happened (entity type, decision, reason), not the sensitive value itself. |
| **Consent** | **v0.1 (deployed): real, not just planned.** `POST /api/scan` fails closed with a `CONSENT_REQUIRED` error for any user who hasn't accepted their organisation's current `consent_policy_version` — checked fresh against the database on every call, not read from a JWT that can't reflect a policy change or an acceptance that happened after the token was issued (`backend/src/routes/scan.rs`, `backend/src/routes/consent.rs`). The browser extension's popup shows a plain-language consent screen (what's scanned, redacted, and logged) before a new user can do anything; `POST /api/consent/accept` records the acceptance with a timestamp and the specific policy version accepted, rejecting a stale version rather than silently recording it. Every user seeded before this existed (including the AI4I demo account) was backfilled to already-consented, so this is a real gate for new users/organisations, not a retroactive disruption of the judged demo — see [Questions.md](../Questions.md). |
| **Access control** | **v0.1 (deployed): real, multi-tenant, and role-based** — no longer the narrow "one shared credential" gap this row used to describe. Every data-bearing API endpoint (`/api/scan`, `/api/audit-log`, `/api/fairness`, `/api/drift`, `/api/security-events`, `/api/command-center/summary`, `/api/health-data-guard/summary`) requires a valid JWT AND filters every query by the caller's `organisation_id`, with no exceptions — verified by dedicated cross-tenant isolation tests (`backend/tests/multi_tenant_isolation.rs`) that create a second organisation and confirm zero rows leak across the boundary, not just that the right rows come back. Three roles (`staff`, `department_reviewer`, `compliance_admin`) gate *what* a user can see within their own organisation: `staff` gets `/api/scan` only, `department_reviewer` gets the audit log scoped to their own department, `compliance_admin` sees everything within their organisation, never across organisations (`auth::require_role` — see `backend/src/auth.rs`). Any organisation can self-register via `POST /api/organisations/signup`. The dashboard *page* itself (HTML/JS) is still public at the demo URL, same as any public web page; what's actually access-controlled is the data it fetches. **A real login page now exists** (`/login`, native chat feature), calling the same auth endpoint — a `staff` login is redirected to `/chat` since `require_role` already denies that role every dashboard-reading endpoint; `compliance_admin`/`department_reviewer` reach the full dashboard. This is opt-in today, not a hard gate: visiting `/` directly without using `/login` first still falls back to the fixed demo account, same as before. **Target**: the same model extended so every dashboard route requires a real session (not just `/chat`), plus a self-service signup screen to match the backend endpoint that already exists. |
| **Authentication** | **v0.1 (deployed)**: real. JWT session tokens (`jsonwebtoken` crate, now carrying `organisation_id` alongside department/role) issued after Argon2-verified password login (`argon2` crate) — see `backend/src/auth.rs` and `backend/src/routes/auth.rs`. Verified live: `POST /api/auth/login` against the deployed Render backend returns a valid token for correct credentials and `401` otherwise. New organisations get real, distinct per-user credentials through `POST /api/organisations/signup` today, not just a shared seeded account — see the Access control row above. **Target**: the same JWT/Argon2/multi-tenant foundation, with a dashboard UI in front of the login and signup endpoints that already exist. |
| **Secrets management** | **v0.1 (deployed)**: real secrets (the JWT signing secret, the database connection string) are never committed. **An organisation's OpenAI API key (native chat feature) is encrypted at rest with AES-256-GCM** (`backend/src/crypto.rs`, `organization_api_keys.encrypted_key`) — a fresh random nonce per encryption, keyed by a required `API_KEY_ENCRYPTION_KEY` environment variable never committed to the repo. This is the first reversible-encryption pattern this codebase has needed — every prior secret-like value (user passwords) uses one-way Argon2 hashing instead, which is unsuitable here since the real key must later be decrypted to actually call OpenAI. The real key is never returned by any API response once saved, only a masked `last_four` confirmation. — `render.yaml` uses `sync: false` for `JWT_SIGNING_SECRET`/`CORS_ORIGIN` so Render prompts for them in the Dashboard rather than storing them in the file, and `DATABASE_URL` is sourced via `fromDatabase` rather than hardcoded. `.env`/`.env.local` are git-ignored; only `.env.example`-style placeholder files are committed. **One intentional, explicit exception**: the seeded demo account's password (`LangoDemo123!`, for `compliance@lango.demo`) *is* committed in the open (`backend/src/bin/seed.rs`, README.md, `.env.local.example`) and is also compiled directly into the frontend's client-side JS bundle via `NEXT_PUBLIC_DEMO_PASSWORD` — meaning it's trivially visible to anyone inspecting the deployed site's network traffic or source, not just anyone reading this repo. This is a deliberate, low-risk decision, not an oversight: that credential only ever grants access to synthetic, seed-script-generated data belonging to one specific demo organisation ("Regional Commercial Bank Demo"), with no real institutional value, and rotating it costs nothing (rerun `cargo run --bin seed`). Kept deliberately unchanged by the multi-tenancy change specifically for continuity with the already-submitted AI4I materials, which reference this exact account — see [Questions.md](../Questions.md). It should not be read as a violation of "never commit credentials" — it's a scoped, documented exception for a demo account with nothing real behind it, and every *other* organisation created since (via `POST /api/organisations/signup`) has its own real, non-public credentials. **Target**: AI provider keys and per-institution database credentials managed via environment variables / a secrets manager — the per-user, per-organisation credential model itself is already real, this demo account is just one deliberately-public account within it, not a stand-in for the whole system's access model anymore. |
| **Rate limiting & dependency hardening** | **v0.1 (deployed)**: a global, per-IP rate limit (`tower_governor`, `backend/src/rate_limit.rs`) now sits as the outermost layer on the entire router — 10 requests/second sustained, burst of 30, keyed on the caller's IP via `SmartIpKeyExtractor` (reads `X-Forwarded-For`/`X-Real-IP`/`Forwarded`, trustworthy specifically because this backend is deployed behind Render's own reverse proxy, not exposed directly). It covers every route by construction, including `/api/auth/login` (previously unprotected against credential-stuffing) and `/api/scan`/`/api/scan/response` (previously unprotected against spam/abuse loops), because it's applied once to the whole `Router` rather than opted into per-endpoint — so no route added by an earlier task, and no route added after this one, can accidentally ship without it. Verified with real tests that fire actual HTTP requests through the real layer (`cargo test --lib rate_limit`), not just a config assertion. A dependency-vulnerability pass (`cargo audit`, `npm audit`) was also run: every npm `high`-severity finding was fixed; of `cargo audit`'s findings, one HIGH-severity issue (`lopdf`, pulled in transitively by the PDF-export feature's `printpdf` dependency) could not be safely fixed without a breaking upstream migration and is deliberately deferred — see [Questions.md](../Questions.md) for the full reasoning on both. **This was an internal hardening pass, not a penetration test** — a real, paid or dedicated external pentest remains explicit future work, not something this pass claims to replace. **Target**: per-route/per-organisation rate limiting tuned to each endpoint's actual traffic pattern, and the deferred `lopdf`/`sqlx` dependency upgrades once their breaking-change risk can be properly verified against a live environment. |
| **Encryption** | **v0.1 (deployed)**: real HTTPS in transit on both ends — Vercel's default TLS termination for the frontend, Render's default TLS termination for the backend API, and TLS to the managed Postgres instance (`sqlx`'s `runtime-tokio-rustls` feature; see `backend/Dockerfile`'s `ca-certificates` install). Encryption at rest relies on Render's platform-level guarantees for its managed Postgres — Lango doesn't implement its own database encryption-at-rest layer, and that hasn't been independently verified against Render's documentation. **Target**: same transit encryption, plus a verified encryption-at-rest posture (whether via the hosting platform or Lango's own layer) formally confirmed for the pilot's data-residency and compliance requirements, given the sensitivity of what the audit log indirectly reflects. |
| **Auditability** | **v0.1 (deployed)**: real. `GET /api/audit-log` serves an actual permanent, queryable, paginated record (user, timestamp, department, entities detected, risk score, decision, reason, model used, response-scan result) written by `/api/scan` on every call — not an illustration. Content is synthetic (seed-script-generated prompts, or manual test prompts), but the write path, storage, and retrieval are real. **The response-scan result column is now real, not a placeholder**, on any row whose prompt was actually sent: the browser extension captures the AI's reply once it renders and submits it to `POST /api/scan/response` (chatgpt.com, claude.ai, gemini.google.com), which updates the same row — see [ARCHITECTURE.md](ARCHITECTURE.md)'s dedicated Response scanning section for the full design, including why a flagged response is only ever flagged, never modified. **A structured export for regulator/internal-audit review now exists**: `GET /api/compliance-export` (`compliance_admin` only) produces a date-ranged CSV (complete dataset) or PDF (readable summary) covering the audit log, fairness metrics, and drift history together — see [Questions.md](../Questions.md) item 24. **Target**: same record shape and export mechanism at institutional volume and retention. |
| **Human oversight** | **v0.1 (deployed)**: real, and running on live-computed data, not illustrative figures. Lango fails closed on system/network failure and on near-zero-confidence matches — including every low-confidence match on a structured entity type (national ID, bank account, phone number, credit card, medical record number, API key), which still blocks (`blocked_low_confidence`) with no exception. For low-confidence name matches specifically, Lango redacts and forwards automatically rather than blocking, logging the action for asynchronous compliance review (`redacted_low_confidence_review`), since blocking on every uncertain name match created unacceptable workflow friction relative to the actual risk — a deliberate, reasoned tradeoff, not a weakening of fail-closed for the cases that actually matter. Real logic in `backend/src/detection/scan.rs`, verified via `cargo test` and live `curl` testing. A fairness threshold breach (DIR below 0.80) is computed live from real `audit_log` rows on every `/api/fairness` request (`backend/src/routes/fairness.rs`) — not a fixed demo number. A drift alert (PSI above 0.20) is real PSI/KL math (`backend/src/detection/drift.rs`), though only computed once at seed time rather than by a continuous live job (see docs/DATA_AI_USAGE.md). In all three cases the system flags rather than auto-corrects — no code path silently updates a rule or dismisses an alert. **The human review loop now closes**: a `compliance_admin`/`department_reviewer` confirming or overturning a flagged low-confidence row (`POST /api/audit-log/:id/review-decision`) is captured as a labelled example (`review_decisions` table, migration 0014), exportable for future rule-tuning — deliberately capture-only, nothing retrains or fine-tunes automatically from it. **Target**: same three flagging mechanisms, with drift also running on a real recurring schedule against live institutional traffic, and the labelled-example dataset actually feeding a real rule-tuning process (still future work, not this task). **Response scanning — both the extension's and native chat's — is deliberately fail-open, not fail-closed, and this is a different rule from every other row above, for a specific, stated reason**: fail-closed exists to stop sensitive data from ever leaving the organisation, and a response has already streamed to the user's screen by the time any scan could run — there is nothing left to stop. A flagged response is never blocked, modified, or hidden; it is only retroactively flagged (a warning banner in the extension, `chat_messages.response_flagged`/an amber warning in the native chat UI), and a scan failure (a network error, an unreachable backend) just means no warning appears, not a security gap the way a failed prompt scan would be — see [ARCHITECTURE.md](ARCHITECTURE.md)'s Response scanning and Native chat sections for the full reasoning. |
| **Misuse risk** | Lango sits in a position of real trust — it processes (in the target system) the full, unredacted content of every staff prompt before redaction happens, meaning a compromise of Lango itself would expose exactly the sensitive data it's meant to protect. This is a concentration-of-risk tradeoff inherent to the gateway model, and is why tenant isolation, encryption, and access control are treated as first-class requirements rather than an afterthought — tenant isolation specifically is no longer just a stated intention: it's real, query-level, and tested (see Access control above). The deployed v0.1 narrows this risk today only because it carries no real institutional data yet — the architecture-level exposure the gateway model implies is unchanged and not something v0.1 has "solved." |
| **Bias and fairness** | Actively monitored, not assumed absent, and now computed live rather than illustrated: the Fairness Audit view calls a real backend endpoint (`GET /api/fairness`) that computes Disparate Impact Ratio and Statistical Parity Difference by language and department from actual `audit_log` rows on every request. The specific numbers shown will vary between deployments and reseeds (they are no longer a single fixed "worked example" like DIR 0.67), but the mechanism — surfacing a DIR-below-0.80 breach as a mandatory review trigger rather than hiding it — is real, tested code, not a scripted demo moment. |

## Stigma-aware aggregate reporting (health module)

Built for the Cimas Healthathon 3.0 submission (see [HEALTH_MODULE.md](HEALTH_MODULE.md) for
the full, self-contained writeup) — this is a real design principle, not a cosmetic choice, so
it's stated here plainly alongside the rest of this document's security/privacy posture.

Every `special_category_health` detection (diagnosis codes, medication names, medical aid
numbers, lab result values, next-of-kin names — see `backend/src/detection/health_rules.rs`)
carries a distinct, additional risk beyond ordinary PII exposure: **re-identification through
aggregation.** A per-department or per-week breakdown by specific condition or medication type
— even with no names attached, even if every individual number looks harmless in isolation —
can be enough to identify who a detection came from once a group is small enough. In the
Zimbabwean context this module was built for, **HIV-status stigma specifically** is the concrete
risk this guards against: a handful of diagnosis-code detections attributed to one small team
over one short window can quietly point back to a specific person, and the harm of that
(discrimination, social exclusion) is categorically worse than an ordinary PII leak. This does
not require anyone to act maliciously — it can happen from entirely ordinary, well-intentioned
dashboard use by a compliance officer who never intended to identify anyone.

Because of this, the rule is applied structurally, not left to reviewer discipline:

- **Restricted** — any aggregate or trend surface: `GET /api/health-data-guard/summary` and the
  Health Data Guard dashboard view return only a total `special_category_health` count, a
  redaction rate, and the coarse `standard`/`special_category_health` split, plus a
  facility-type parity comparison. They never group or filter by the specific entity type
  detected (diagnosis_code vs. medication_name vs. etc.), let alone by the specific ICD-10 code
  or medication name decoded from a match — see `backend/src/routes/health.rs`'s own comment,
  which carries the identical reasoning next to the code it governs.
- **Not restricted** — the existing, per-entry Audit Log detail view (the existing expandable
  row, scoped to one specific, already-flagged session). A compliance officer reviewing that one
  row can still see exactly which entity types were detected in it (`AuditLogRow.sensitivity_class`,
  `routes::audit_log::get_audit_log`, both unrestricted, both scoped to one row at a time) — that
  is the legitimate, authorized, per-case review this product exists to support. The restriction
  is specifically on *aggregate* and *trend* reporting, not on that existing per-entry detail.

## Risk-level classification

**Medium-to-high risk.** This is stated plainly, not softened: Lango is designed to
sit in front of institutional AI usage handling health, financial, and national
identity data (national IDs, bank account numbers, medical record numbers, phone
numbers) at banks, hospitals, and ministries. The fact that the mechanism is
specifically designed to redact and protect this data does not lower the risk
classification of the system itself — a gateway positioned to see this category of
data before redaction, and to hold the audit trail of when and where it was flagged,
is inherently a high-value target and a system whose failure modes (a missed
detection, a compromised gateway, a fairness gap in production) have real
institutional and individual consequences. A real backend and detection engine now
exist and are publicly deployed (Render + Vercel — see [ARCHITECTURE.md](ARCHITECTURE.md)),
which is a meaningfully different exposure than an offline demo: the live login and
API endpoints are reachable by anyone, not just a developer's own machine. In
practice, that exposure is still bounded — no live AI provider connection, no real
institutional data processed, and a basic internal hardening pass (rate limiting,
dependency audit, credential-handling review — see the Rate limiting & dependency
hardening row above) rather than a formal external penetration test — but it is not zero the
way a purely local, unreachable build would be, which is why the demo organisation's
shared credential is called out explicitly above as a scoped exception rather than
left implicit — it is now one deliberately-public account among what could be many
real, isolated organisations, not a description of the whole system's access model.
The classification above describes the target system this submission is
proposing to build, and is assigned accordingly rather than claiming "low risk"
because today's build, though real and deployed, is still v0.1 and not yet handling
anything of real value.
