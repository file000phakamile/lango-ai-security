# Security and Privacy — Lango / AI Data Guard

| Area | Lango's answer |
|---|---|
| **Data minimisation** | **v0.1 (deployed)**: no real institutional data is collected — every prompt processed today is synthetic (seed-script-generated or manually tested), and the system never stores raw prompt text regardless: `backend/src/detection/scan.rs` redacts matched entities before anything is logged, and `audit_log.original_prompt_hash` stores only a SHA-256 hash of the original, never the text itself. This is real, running code, not a target-state description. **Target**: same principle at institutional scale — the Redaction Engine strips sensitive entities before a prompt is forwarded to an AI provider; the audit log records that a redaction happened (entity type, decision, reason), not the sensitive value itself. |
| **Consent** | **v0.1 (deployed)**: not applicable — the only people hitting the live system today are testers and reviewers, not real institutional staff, and no real personal data is processed. **Target**: the pilot checklist (Pilot & Sandbox view) includes "data-use consent flow signed off" as a precondition before any pilot user is onboarded; staff would need to be informed that their AI-tool prompts are scanned and logged before using the gateway. |
| **Access control** | **v0.1 (deployed)**: real, but narrow. Every data-bearing API endpoint (`/api/scan`, `/api/audit-log`, `/api/fairness`, `/api/drift`, `/api/security-events`, `/api/command-center/summary`) requires a valid JWT, enforced server-side (`auth::AuthUser`, `auth::require_role` — see `backend/src/auth.rs`); an unauthenticated request gets a real `401`, confirmed by direct testing. The gap: there's only one seeded demo account today (see Secrets management below) and no login UI, so "access control" currently means "one shared credential," not per-user authorization — a v0.1 shortcut, not the target model. The dashboard *page* itself (HTML/JS) is still public at the demo URL, same as any public web page; what's actually access-controlled is the data it fetches. **Target**: role-based access (staff who submit prompts vs. compliance/admin roles who view dashboards and manage pattern rules) for real, distinct users, scoped per institution (tenant isolation). |
| **Authentication** | **v0.1 (deployed)**: real. JWT session tokens (`jsonwebtoken` crate) issued after Argon2-verified password login (`argon2` crate) — see `backend/src/auth.rs` and `backend/src/routes/auth.rs`. Verified live: `POST /api/auth/login` against the deployed Render backend returns a valid token for correct credentials and `401` otherwise. **Target**: same JWT/Argon2 foundation, extended to real per-user credentials issued through a real signup/provisioning flow instead of one shared seeded account. |
| **Secrets management** | **v0.1 (deployed)**: real secrets (the JWT signing secret, the database connection string) are never committed — `render.yaml` uses `sync: false` for `JWT_SIGNING_SECRET`/`CORS_ORIGIN` so Render prompts for them in the Dashboard rather than storing them in the file, and `DATABASE_URL` is sourced via `fromDatabase` rather than hardcoded. `.env`/`.env.local` are git-ignored; only `.env.example`-style placeholder files are committed. **One intentional, explicit exception**: the seeded demo account's password (`LangoDemo123!`, for `compliance@lango.demo`) *is* committed in the open (`backend/src/bin/seed.rs`, README.md, `.env.local.example`) and is also compiled directly into the frontend's client-side JS bundle via `NEXT_PUBLIC_DEMO_PASSWORD` — meaning it's trivially visible to anyone inspecting the deployed site's network traffic or source, not just anyone reading this repo. This is a deliberate, low-risk decision, not an oversight: that credential only ever grants access to synthetic, seed-script-generated data with no real institutional value, and rotating it costs nothing (rerun `cargo run --bin seed`). It should not be read as a violation of "never commit credentials" — it's a scoped, documented exception for a demo account with nothing real behind it. **Target**: AI provider keys and per-institution database credentials managed via environment variables / a secrets manager, with real per-user credentials replacing the shared demo account entirely — this exception would not exist in a real deployment. |
| **Encryption** | **v0.1 (deployed)**: real HTTPS in transit on both ends — Vercel's default TLS termination for the frontend, Render's default TLS termination for the backend API, and TLS to the managed Postgres instance (`sqlx`'s `runtime-tokio-rustls` feature; see `backend/Dockerfile`'s `ca-certificates` install). Encryption at rest relies on Render's platform-level guarantees for its managed Postgres — Lango doesn't implement its own database encryption-at-rest layer, and that hasn't been independently verified against Render's documentation. **Target**: same transit encryption, plus a verified encryption-at-rest posture (whether via the hosting platform or Lango's own layer) formally confirmed for the pilot's data-residency and compliance requirements, given the sensitivity of what the audit log indirectly reflects. |
| **Auditability** | **v0.1 (deployed)**: real. `GET /api/audit-log` serves an actual permanent, queryable, paginated record (user, timestamp, department, entities detected, risk score, decision, reason, model used, response-scan result) written by `/api/scan` on every call — not an illustration. Content is synthetic (seed-script-generated prompts, or manual test prompts), but the write path, storage, and retrieval are real. **Target**: same record shape at institutional volume and retention, with a structured export (CSV/JSON) for regulator or internal-audit review — not yet built. |
| **Human oversight** | **v0.1 (deployed)**: real, and running on live-computed data, not illustrative figures. Lango fails closed on system/network failure and on near-zero-confidence matches — including every low-confidence match on a structured entity type (national ID, bank account, phone number, credit card, medical record number, API key), which still blocks (`blocked_low_confidence`) with no exception. For low-confidence name matches specifically, Lango redacts and forwards automatically rather than blocking, logging the action for asynchronous compliance review (`redacted_low_confidence_review`), since blocking on every uncertain name match created unacceptable workflow friction relative to the actual risk — a deliberate, reasoned tradeoff, not a weakening of fail-closed for the cases that actually matter. Real logic in `backend/src/detection/scan.rs`, verified via `cargo test` and live `curl` testing. A fairness threshold breach (DIR below 0.80) is computed live from real `audit_log` rows on every `/api/fairness` request (`backend/src/routes/fairness.rs`) — not a fixed demo number. A drift alert (PSI above 0.20) is real PSI/KL math (`backend/src/detection/drift.rs`), though only computed once at seed time rather than by a continuous live job (see docs/DATA_AI_USAGE.md). In all three cases the system flags rather than auto-corrects — no code path silently updates a rule or dismisses an alert. **Target**: same three mechanisms, with drift also running on a real recurring schedule against live institutional traffic. |
| **Misuse risk** | Lango sits in a position of real trust — it processes (in the target system) the full, unredacted content of every staff prompt before redaction happens, meaning a compromise of Lango itself would expose exactly the sensitive data it's meant to protect. This is a concentration-of-risk tradeoff inherent to the gateway model, and is why tenant isolation, encryption, and access control are treated as first-class requirements rather than an afterthought. The deployed v0.1 narrows this risk today only because it carries no real institutional data yet — the architecture-level exposure the gateway model implies is unchanged and not something v0.1 has "solved." |
| **Bias and fairness** | Actively monitored, not assumed absent, and now computed live rather than illustrated: the Fairness Audit view calls a real backend endpoint (`GET /api/fairness`) that computes Disparate Impact Ratio and Statistical Parity Difference by language and department from actual `audit_log` rows on every request. The specific numbers shown will vary between deployments and reseeds (they are no longer a single fixed "worked example" like DIR 0.67), but the mechanism — surfacing a DIR-below-0.80 breach as a mandatory review trigger rather than hiding it — is real, tested code, not a scripted demo moment. |

## Risk-level classification

**Medium-to-high risk.** This is stated plainly, not softened: Lango is designed to
sit in front of institutional AI usage handling health, financial, and national
identity data (national IDs, bank account numbers, medical record numbers, phone
numbers) at banks, hospitals, and ministries. The fact that the mechanism is
specifically designed to redact and protect this data does not lower the risk
classification of the system itself — a gateway positioned to see this category of
data before redaction, and to hold the audit trail of when and where it was flagged,
is inherently a high-value target and a system whose failure modes (a missed
detection, a compromised gateway, a fairness gap in production) have real
institutional and individual consequences. A real backend and detection engine now
exist and are publicly deployed (Render + Vercel — see [ARCHITECTURE.md](ARCHITECTURE.md)),
which is a meaningfully different exposure than an offline demo: the live login and
API endpoints are reachable by anyone, not just a developer's own machine. In
practice, that exposure is still bounded — no live AI provider connection, no real
institutional data processed, no production hardening pass — but it is not zero the
way a purely local, unreachable build would be, which is why the single shared demo
credential is called out explicitly above as a scoped exception rather than left
implicit. The classification above describes the target system this submission is
proposing to build, and is assigned accordingly rather than claiming "low risk"
because today's build, though real and deployed, is still v0.1 and not yet handling
anything of real value.
